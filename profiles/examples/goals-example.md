"Help them ship" means: users point to Pixel Forge as the thing that made their game possible. "Recognized" means: when the asset generation topic comes up on r/gamedev, the name comes up. Impact is the goal; recognition is how I know it's landing.

- [x] Reddit presence established on r/gamedev with at least one post over 100 upvotes
- [x] Discord engagement started in 3+ indie dev servers
- [ ] Game jam sponsorship: provide free Pro access to participants in one mid-size jam

Q1 was product-heavy. The generator needed to work before anything else. Community presence started later than planned but momentum is building.

- [ ] First podcast or interview appearance about indie dev AI tools
- [ ] Monthly retro practice: end-of-month review against signals (what did we learn?)

Signals to watch: balance analyzer activation rate, asset generator to pro upgrade rate, content reach outside r/gamedev.

## Profit

**Annual: Validate the freemium model. Reach $5K MRR by Q3. Run one enterprise or studio experiment.**

Revenue is a signal. $5K MRR doesn't prove product-market fit on its own -- it tells me whether indie devs will pay $19/month and what kind of devs are paying. The data inside that number matters more than the number.

Runway is approximately 18 months of personal living expenses from savings. This is not an emergency, but it's not infinite. Q2 needs a monetization signal, not just engagement.

### Q1 (Jan-Mar)

- [x] Free tier live: real value, real usage (340 registered users in 60 days)
- [x] Pro tier launched at $19/month: 8 paying customers
- [x] Stripe integration working, receipts going out, cancellation flow functional
- [ ] File Q1 estimated taxes (deadline April 15)

Strong start. 8 conversions is a signal but too small to conclude anything. Need 3-5x more to see the conversion pattern.

### Q2 (Apr-Jun)

- [ ] Reach 30 paying Pro subscribers ($570 MRR -- meaningful signal, not meaningful revenue yet)
- [ ] Launch Balance Analyzer with a clear upgrade path for Generator Pro users
- [ ] One direct customer conversation per week: understand why people upgrade and why they don't
- [ ] Define the Q3 monetization experiment: annual plans? Team tier? Studio tier at $99/month?
- [ ] File taxes (if not completed in Q1 estimated)

If Q2 ends without 30 Pro subscribers, something is wrong with either the value prop or the discovery path. "Define the Q3 experiment" is the minimum bar regardless of conversion rate.

## People

**Annual: Be connected to the indie dev community in a way that feels like belonging, not networking.**

Game jams are the social infrastructure here. That's where devs gather, ship fast, and cheer each other on. The question is whether online jam community is enough or if Austin's local game dev scene fills a different need.

### Q1 (Jan-Mar)

- [x] Joined 5 active indie dev Discord servers, contributing (not just lurking)
- [ ] Attended one Austin Game Developers meetup
- [ ] Game jam participation: entered Global Game Jam as a solo dev to build empathy for the product's users

Honest baseline: the professional network is online-first. Most meaningful connections are async in Discord. Local Austin game dev scene is small but active -- worth exploring.

### Q2 (Apr-Jun)

- [ ] Attend 2 Austin game dev meetups -- test if local is valuable or if online community is sufficient
- [ ] Cultivate 3-5 genuine relationships with other indie devs who are further along: people whose games I've played, whose work I respect
- [ ] Balance Analyzer beta users: follow up with each one personally, not just through analytics
- [ ] Game jam: co-develop one jam entry with an artist or another dev (not solo)

Monthly forcing function: at least one real conversation with another indie dev per week. Not a support ticket. Not a Discord lurk. An actual conversation.

## Health

**Annual: Maintain the foundation. Climbing gym is non-negotiable. Game jam burnout is the watch pattern.**

The risk is not general burnout -- it's game jam mode bleeding into the business. Jam mode (ship in 48 hours, sleep later) is fine during jams. Applied to a 12-month product roadmap, it breaks things.

### Q1 (Jan-Mar)

- [x] Climbing gym: 3x/week maintained (the social + physical combo that actually works)
- [x] Therapy: biweekly cadence maintained
- [ ] Gym schedule protected during Balance Analyzer crunch: this slipped in January

Q1 showed the pattern: when a product deadline approaches, the gym is the first thing cut. That's the failure mode to watch.

### Q2 (Apr-Jun)

- [ ] Climbing gym 3x/week with a hard floor: zero weeks below 2x. Track in Terrain.
- [ ] Hard rule: no game jams during Balance Analyzer launch month (May). Jams are a distraction when there's a real launch to support.
- [ ] Identify one more forcing function for rest days -- the gym works because it's social. What else has that property?

The gym-cut pattern during deadlines is a canary. If it slips again in Q2, the scheduling model isn't working and needs a different approach (blocked calendar, accountability partner, something structural).

## How This Doc Works

- **Yearly goals** are the headline for each pillar. Directional but concrete enough to evaluate.
- **Quarterly goals** are specific, checkable targets. Reviewed and updated at the start of each quarter.
- **Weekly goals** live in Terrain.md, not here. Weekly tasks should ladder up to quarterly goals.
- **Monthly direction reviews** (end of each month) are the checkpoint for whether quarterly goals still make sense. If the Balance Analyzer launch moves, the Q2 Profit goals shift. Update the doc.
- **Vehicles vs. pillars:** The Asset Generator, Balance Analyzer, and content pipeline are vehicles that serve the goals. They're tracked in Terrain. This doc stays at the pillar level so the goals survive a pivot. If Pixel Forge pivots away from game assets, the Purpose pillar goal ("help indie devs ship games they couldn't build alone") either updates or stays as the north star that finds a new vehicle.
- **Q3-Q4 goals** set during the mid-year review, informed by Q2 results.
